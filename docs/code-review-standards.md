> **版本**: v1.0

> **适用范围**: ktv-backend / admin-frontend / room-frontend

## 一、审查目标

代码审查的核心目的不是挑毛病，而是：

1. **保障质量** — 在问题到达生产环境之前拦截它
2. **知识共享** — 让团队成员互相学习，提升整体水平
3. **代码一致性** — 让 80 个 Java 文件和 77 个前端文件保持统一的风格和标准
4. **降低维护成本** — 6 个月后回头看代码，仍然能快速理解

---

## 二、审查优先级体系

所有审查意见必须标注优先级，让提交者知道哪些必须改，哪些可以讨论。

| 优先级 | 标记 | 含义 | 处理要求 |
|--------|------|------|----------|
| 🔴 阻塞 | **BLOCKER** | 安全漏洞、数据丢失、编译错误、API 契约破坏 | 必须修复后才能合并 |
| 🟡 建议 | **SHOULD** | 逻辑缺陷、缺失校验、性能问题、缺少测试 | 强烈建议修复，提交者可说明理由暂不处理 |
| 💭 挑剔 | **NITPICK** | 命名改进、代码风格、文档补充、更优方案 | 可选改进，不阻塞合并 |

## 三、后端审查标准（Java / Spring Boot）

### 3.1 🔴 阻塞项（必须修复）

#### 安全性
- **SQL 注入**：禁止拼接 SQL 字符串，必须使用 MyBatis-Plus 的条件构造器或 `#{}` 参数绑定
- **认证绕过**：`/api/admin/**` 路径必须经过 JWT 校验，敏感操作必须有权限控制
- **文件上传漏洞**：
  - 必须校验文件类型（后缀白名单 + MIME 类型双重校验）
  - 必须校验文件大小（已在 application.yml 配置 500MB 上限）
  - 上传路径必须防止目录穿越（禁止 `../`）
  - 文件名必须使用 UUID 重命名，禁止直接使用用户上传的文件名
- **敏感信息泄露**：禁止在日志中打印密码、Token 等敏感信息；生产环境关闭 MyBatis SQL 日志
- **密码处理**：必须使用 BCrypt 加密，禁止明文存储和传输

#### 数据安全
- **Redis Key 规范**：必须遵循 `ktv:{业务}:{标识}` 的命名规范
- **并发安全**：订单状态变更、队列操作等必须考虑并发场景，必要时使用 Redis 分布式锁
- **数据一致性**：Redis 和 MySQL 双写时，必须保证最终一致性
- **事务管理**：涉及多表写操作的方法必须添加 `@Transactional`

#### 接口契约
- **统一返回格式**：所有接口必须使用 `Result<T>` 包装返回值
- **参数校验**：Controller 层接收的 DTO 必须使用 `@Valid` / `@Validated` 注解
- **异常处理**：业务异常必须抛出 `BusinessException`，禁止在 Controller 层 try-catch 吞掉异常
- **分页参数**：统一使用 `current` / `size`（不是 pageNum / pageSize）

### 3.2 🟡 建议项（应该修复）

#### 正确性
- **NPE 防御**：所有 DTO 字段读取前必须做 null 检查（参考历史 Bug D3/D4/D5/E1/E2）
- **update 逻辑**：更新操作中，null 字段应保留原值，不应覆盖（参考 Bug 7）
- **类型安全**：Redis 操作应使用 `StringRedisTemplate`，避免序列化问题（参考 Bug 2/20）

#### 可维护性
- **分层职责**：
  - Controller：只做参数接收和转发，不含业务逻辑
  - Service：核心业务逻辑，可调用 Mapper 和其他 Service
  - Mapper：只做数据访问，不含业务判断
- **命名规范**：
  - 方法名应清晰表达意图（`findAvailableRooms` 优于 `getRooms`）
  - 避免缩写（`singerName` 优于 `sName`）
- **代码重复**：超过 3 处相同逻辑应提取为公共方法

#### 性能
- **N+1 查询**：避免循环中调用数据库，使用批量查询或 JOIN
- **Redis 缓存穿透**：对不存在的数据也应缓存空值或使用布隆过滤器
- **大事务**：事务范围应尽量小，避免在事务中调用外部接口

### 3.3 💭 挑剔项（最好有）

- **Javadoc**：公共 API 应有 Javadoc 注释，包括 `@param` / `@return` / `@throws`
- **日志规范**：使用 `@Slf4j` + 占位符 `{}`，禁止字符串拼接
- **魔法值**：状态码、常量应定义为 `enum` 或 `static final` 常量
- **Lombok 使用**：实体类统一使用 `@Data`，Service 统一使用 `@RequiredArgsConstructor`
- **作者标注**：所有文件 `@author` 统一为 `shaun.sheng`

## 四、前端审查标准（React / JavaScript）

### 4.1 🔴 阻塞项（必须修复）

#### 安全性
- **XSS 防护**：禁止使用 `dangerouslySetInnerHTML`（当前项目已规避 ✅）
- **Token 存储**：JWT Token 存储在 localStorage，不得存放在 Cookie 中
- **API 请求**：必须通过封装的 `request.js` 发送请求，禁止直接使用 axios
- **接口隔离**：
  - admin-frontend 只能调用 `/api/admin/**`
  - room-frontend 只能调用 `/api/room/**` 和 `/api/media/**`
  - 严禁跨端调用（参考致命 Bug 19：Join 页面调用了 admin 接口）

#### 数据处理
- **分页参数**：前端传递 `pageNum` / `pageSize`，需确认后端 `current` / `size` 的映射关系
- **状态管理**：orderId 等关键状态必须通过 Zustand persist 持久化
- **错误处理**：API 调用必须有 catch 处理，禁止未捕获的 Promise rejection

### 4.2 🟡 建议项（应该修复）

#### React 最佳实践
- **Hooks 规则**：
  - 自定义 Hook 必须以 `use` 开头
  - useEffect 的依赖数组必须完整，避免闭包陷阱（参考 Bug R6）
  - 回调函数传入子组件时应使用 `useCallback`（参考 Bug R4）
- **定时器清理**：所有 `setTimeout` / `setInterval` 必须在 useEffect 清理函数中清除（参考 Bug R3/R5）
- **状态引用**：在异步回调中访问 state 时，应使用 `useRef` 保持最新值（参考 Bug R6）

#### 组件设计
- **单一职责**：一个组件文件超过 300 行应考虑拆分
- **Props 校验**：复杂组件应定义 PropTypes 或迁移到 TypeScript
- **Key 使用**：列表渲染的 key 必须使用唯一标识符，禁止使用数组索引

#### 性能
- **懒加载**：路由组件必须使用 `React.lazy` + `Suspense`
- **避免不必要的渲染**：频繁更新的状态应考虑使用 `useMemo` / `useCallback`
- **图片优化**：封面图等静态资源应使用合理的尺寸和格式

### 4.3 💭 挑剔项（最好有）

- **CSS 模块化**：组件样式应使用独立的 CSS 文件，命名用 BEM 规范
- **语义化**：使用 `<nav>` / `<main>` / `<section>` 等语义标签
- **无障碍**：按钮和交互元素应有 `aria-label`
- **代码组织**：工具函数提取到 `utils/`，常量提取到 `constants/`

## 五、数据库审查标准

### 🔴 阻塞项
- **表结构一致性**：所有表必须有 `id`、`create_time`、`update_time`、`deleted` 字段
- **逻辑删除**：必须使用 `deleted` 字段（0=未删除，1=已删除），禁止物理删除
- **字段命名**：统一下划线命名（snake_case），表名统一 `t_` 前缀
- **索引合理**：高频查询字段必须有索引，避免全表扫描
- **字符集**：统一使用 `utf8mb4`，支持 emoji

### 🟡 建议项
- **SQL 可读性**：复杂 SQL 应有注释说明业务含义
- **字段长度**：VARCHAR 长度应合理，避免过度分配
- **时间字段**：使用 `DATETIME` 类型，统一存储服务器时间

## 六、审查流程

### 6.1 提交前自检清单

在发起代码审查之前，提交者应完成以下自检：

```
□ 代码已通过本地编译（mvn compile / npm run build）
□ 前端已通过 ESLint 检查（npm run lint）
□ 新增/修改的接口已在本地测试通过
□ 没有引入新的 TODO/FIXME/HACK（或已记录 Issue）
□ 没有提交敏感信息（密码、密钥、Token）
□ commit message 清晰描述了改动内容
□ 没有提交调试代码（console.log / System.out.println）
```

### 6.2 审查流程

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐     ┌──────────┐
│  提交代码    │ ──▶ │  自动化检查   │ ──▶ │  人工审查    │ ──▶ │  合并代码  │
└─────────────┘     └──────────────┘     └─────────────┘     └──────────┘
                          │                    │
                    ┌─────┴─────┐        ┌─────┴─────┐
                    │ 编译检查   │        │ 正确性     │
                    │ Lint 检查  │        │ 安全性     │
                    │ 单元测试   │        │ 可维护性   │
                    └───────────┘        │ 性能       │
                                         └───────────┘
```

### 6.3 审查轮次管理

| 轮次 | 关注点 | 检查方法 |
|------|--------|----------|
| 第一轮 | **正确性 + 安全性** | 通读代码，关注 🔴 阻塞项 |
| 第二轮 | **可维护性 + 性能** | 深入逻辑，关注 🟡 建议项 |
| 第三轮 | **代码风格 + 文档** | 快速扫视，关注 💭 挑剔项 |

### 6.4 审查响应时效

| 优先级 | 响应时限 |
|--------|----------|
| 🔴 阻塞 | 发现后立即处理，不超过 2 小时 |
| 🟡 建议 | 在当前迭代内处理 |
| 💭 挑剔 | 记录到 TODO，后续迭代处理 |

## 七、审查意见格式模板

每条审查意见应遵循以下格式，确保提交者能清晰理解：

```
[优先级标记] **问题类型：简短描述**

**位置**：`文件路径:行号`

**问题**：具体说明哪里出了什么问题

**为什么**：解释为什么这是一个问题（帮助提交者理解原理）

**建议**：
\`\`\`java/javaScript
// 修复前（如有）
problematic code

// 修复后（如有）
suggested fix
\`\`\`
```

### 示例

```
🔴 **安全：SQL 注入风险**

**位置**：`SongSearchServiceImpl.java:45`

**问题**：用户输入的 keyword 直接拼接到了 SQL LIKE 条件中：
\`\`\`java
queryWrapper.like("name", keyword);
\`\`\`

**为什么**：如果 keyword 包含 `%`、`_` 等 LIKE 通配符，可能导致非预期的查询结果。
虽然 MyBatis-Plus 的 `like()` 方法使用了参数绑定不会造成注入，
但 LIKE 通配符仍需转义。

**建议**：
\`\`\`java
// 转义 LIKE 通配符
String safeKeyword = keyword.replace("%", "\\%").replace("_", "\\_");
queryWrapper.like("name", safeKeyword);
\`\`\`
```

## 九、自动化审查建议（下一步）

## 十、附录

### A. 审查检查清单（Checklist）

可复制使用，用于每次审查：

```
## 代码审查 - [PR标题]

### 正确性
- [ ] 业务逻辑是否正确？
- [ ] 边界条件是否处理？
- [ ] null/空值/异常情况是否防御？

### 安全性
- [ ] 是否存在注入风险？
- [ ] 认证/授权是否正确？
- [ ] 敏感信息是否泄露？
- [ ] 文件上传是否安全？

### 可维护性
- [ ] 命名是否清晰？
- [ ] 代码是否容易理解？
- [ ] 是否有过多的代码重复？
- [ ] 分层是否合理？

### 性能
- [ ] 是否有 N+1 查询？
- [ ] 是否有不必要的资源浪费？
- [ ] 缓存使用是否合理？

### 测试
- [ ] 是否有对应的测试？
- [ ] 测试是否覆盖关键路径？
- [ ] 测试是否可读可维护？

### 审查结论
- [ ] ✅ 通过 — 无阻塞问题
- [ ] 🔄 需修改 — 有阻塞/建议项需处理
- [ ] ❌ 驳回 — 有严重问题需要重做
```

### B. Commit Message 规范

```
<type>(<scope>): <subject>

<body>
```

**type 类型**：
- `feat`: 新功能
- `fix`: 修复 Bug
- `refactor`: 重构（不改变行为）
- `style`: 代码格式调整
- `docs`: 文档更新
- `test`: 测试相关
- `chore`: 构建/工具变动

**scope 范围**：
- `backend`: 后端
- `admin`: 后台管理前端
- `room`: 包厢点歌前端
- `sql`: 数据库
- `infra`: 基础设施（CI/CD、配置等）

**示例**：
```
fix(backend): SongServiceImpl.updateSong 空指针防御

当 name 为 null 时, equals 和 PinyinHelper.toPinyin 会抛出 NPE。
添加 null 检查保护。

Closes #issue-number
```
